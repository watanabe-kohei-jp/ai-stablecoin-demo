## 1. 基礎：ステーブルコインとは

**法定通貨に1:1で価値を固定したデジタル通貨**。価格が動かないこと（stable）が存在価値。

| 裏付けの型 | 仕組み | 代表例 |
|---|---|---|
| **法定通貨担保型** | 1枚=1ドル等の現金・国債を準備金として保有 | USDC, USDT, JPYC |
| 暗号資産担保型 | 暗号資産を過剰担保にして発行 | DAI |
| 無担保（アルゴリズム）型 | アルゴリズムで需給調整（**破綻例あり**＝UST/Terra） | （非推奨） |

### なぜ AI と相性が良いのか

| SC の性質 | AI にとっての意味 |
|---|---|
| 価格が動かない | 「いくら払うか」を安定して計算できる |
| プログラマブル | 条件付き支払い・上限・自動執行をコードで書ける |
| 24/365・即時・国境なし | 機械速度（ミリ秒）の取引に銀行営業時間が邪魔をしない |
| 手数料ほぼゼロ・極小額OK | $0.01 単位のマシン間決済が成立（カードは手数料下限~$0.30で不可） |

> 一言：**SC は「AI が扱える形をしたお金」**。だから AI 時代の決済インフラ候補No.1。

## 2. 技術：x402 はどう動くか

**x402** = HTTP の「402 Payment Required」（昔から予約されていた未使用ステータス）を復活させ、
API アクセスをステーブルコインで即時決済する仕組み（Coinbase + Cloudflare 発）。

### 決済の1往復

```
買い手AI                                   売り手API
 ① GET /search → ② 402 Payment Required（料金・通貨・宛先・network）
 ③ ウォレットが「支払い認可」に署名（EIP-3009 = ガスレスな送金指示）
 ④ X-PAYMENT を付けて同じGETを再送 → ⑤ facilitator が署名検証＆オンチェーン決済
 ⑥ 200 OK + データ + 決済レシート（tx hash）
```

### キーとなる3つの部品

| 部品 | 役割 |
|---|---|
| **EIP-3009**（`transferWithAuthorization`） | 「この額をこの宛先に送ってよい」という**署名だけ**を作る。買い手はガス(ETH)を払わない |
| **facilitator** | 署名を検証し、**自分がガスを立て替えて**オンチェーン送金を実行する代行サービス。資金は預からない |
| **resource server**（売り手） | 料金条件を 402 で提示するだけ。**鍵を持たず署名しない** |

> だから買い手は ETH 残高ゼロでも支払える（ガスは facilitator 持ち）。本デモでも実証済み。

### このデモでの対応

- network：Base Sepolia（`eip155:84532`）/ 通貨：テストUSDC（`0x036C…`, decimals=6）
- 売り手 `server/shop.py` が `exact` スキームで保護、買い手 `agent/wallet.py` が EIP-3009 署名
- 「払う前=402／払った後=200」を実際のトランザクションで確認できる

## 前提補正：号数（資産の性質）× 発行体（免許）の2軸で語る

日本のステーブルコイン＝改正資金決済法上の「**電子決済手段**」。よくある「1号/2号＝資金移動業型」という言い方は便宜的すぎる。正確には**2軸**で捉える：

| 軸 | 中身 |
|---|---|
| **号数（資産の性質）** | 1号＝通貨建資産型 / 2号＝1号と相互交換可能なもの / **3号＝特定信託受益権型** / 4号＝これらに準ずる指定型 |
| **発行体（免許）** | 銀行・**資金移動業者**・**信託会社** のいずれか |

- **JPYC** ＝ 資金移動業者が発行する **1号（/2号）型**
- **MUFG系（三菱UFJ信託/Progmat）** ＝ 信託が発行する **3号型（特定信託受益権）**
- 流通を担う業者は「電子決済手段等取引業」として登録制（金融庁、2023/6〜）

> 補足：「号(性質)×発行体(免許)」の2軸で捉えると整理しやすい。

# 未来像：AI × ステーブルコイン（5〜10年）

一言でいえば **「ソフトウェアが予算を持つ時代」**。人間が払うのではなく、AIが
*目的・予算・制約* を与えられ、API・データ・計算資源・他のAIに **都度・極小・条件付き・無人**で支払う。

## 起きる3つの構造変化

| # | 構造変化 | Before → After |
|---|---|---|
| ① | 決済の単位と頻度 | 月額契約・請求書・カード → **利用時の即時・極小（1円未満〜）・無人決済** |
| ② | SCの役割 | 単なる送金手段 → **トークン化資産の「オンチェーンの現金脚（cash leg）」**。DvP/PvP・担保・償還・配当を自動執行 |
| ③ | 銀行の役割 | 「お金を預かる場所」 → **「AIに使わせてよいお金のルールを設計する主体」** |

## 核心の再定義

> AI×SCの本命は「**AIが勝手に買い物する未来**」ではなく、
> 「**企業がAIに限定的な財務権限を委任する未来**」。
> そこで効くのは*コインの発行量*ではなく、**権限管理・監査・信頼・コンプライアンス**。

この再定義が、そのまま三菱UFJの勝ち筋につながる。

# 三菱UFJ（MUFG）への接続

**MUFGは"未来像のピース"をすでに現実に並べ始めている。** 未来図の各パーツと、MUFGの今の一手が対応する。

| 未来像のピース | MUFG／Progmatの現在の一手（2026） |
|---|---|
| ② SCが**RWAの現金脚**になる | トークン化**MMF**（日本初・2026機関投資家向け）、**トークン化株式**（2026春）、**国債**トークン化・24h市場をProgmat主導 |
| ① **法人M2B/越境**の即時決済 | **三菱商事**の200+子会社・国際送金で円SC越境決済PoC。**信用状（LC）並みの支払確実性を銀行保証なしで** |
| ②③ **マルチチェーン相互運用** | ProgmatがCorda→**Avalanche L1へ移行**（2026/6末）、Ethereum/Polygon/Cosmos対応、DvP/PvPを各種SCで（Datachain連携） |

| 信頼レイヤー（発行体） | **三菱UFJ信託**が3号SC（特定信託受益権）の受託者＝信頼の担保 |

# MUFGの勝ち筋と弱点

| | 中身 |
|---|---|
| ✅ 勝ち筋 | 信用・規制対応・**信託スキーム**・法人顧客基盤。主戦場は小売でなく**法人B2B・越境・RWA現金脚**。さらに「**AI決済ガバナンス層**」（誰に・いくらまで・どの条件で払わせるか／上限・許可・承認・監査・AML）は銀行が最強 |
| ⚠️ 弱点 | プロダクト速度、開発者エコシステム、グローバル標準。**世界標準がUSDC/x402前提で固まると、円SCは"国内用のきれいな仕組み"で終わる**リスク |

---

# リスク・反証（楽観に流されない）

| リスク | 中身 |
|---|---|
| 需要そのもの | M2Bマイクロペイメントは過去何度も期待され伸び悩んだ。カード・月額・クラウド課金で足りる可能性 |
| 責任主体 | AI自律決済で誤決済・詐欺・制裁・マネロンが起きたとき「**誰の支払い意思か**」を証明できないと銀行は本格展開しにくい |
| 中抜き | 企業がUSDCを直接使い銀行を通さず決済・運用 → 銀行が**単なる準備資産管理者**に落ちる |
| プロトコル覇権 | USDC/Stripe/Coinbase/Cloudflare/Google が開発者体験と標準を握ると、国内規制に強くても主導権を取れない |

---

## 独自見解（まとめ）

> **MUFGが勝つなら"日本版USDC"ではなく、"法人AI決済の信頼レイヤー"。**
> コイン発行量の競争に行けば海外勢に負ける。だが信託×規制×ガバナンスで
> 「**AIに安心して財務権限を渡せる銀行**」を取れば、日本の銀行の強みが効く。
